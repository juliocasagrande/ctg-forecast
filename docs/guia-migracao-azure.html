⚙️ Guia Técnico

Migração Azure — Passo a Passo

Tudo o que você precisa fazer para deixar o CTG Engineering Forecast rodando novamente na conta corporativa Azure.

⚠️ Por que a aplicação não está acessível?

Ao migrar o App Service de uma conta Azure pessoal para a conta corporativa, nenhuma configuração é transferida automaticamente. O problema não é de permissão de conta — é de variáveis de ambiente e credenciais ausentes no novo ambiente.

Os três bloqueadores principais são: (1) Application Settings (variáveis de ambiente) não configuradas no novo App Service, (2) GitHub Actions secrets apontando para o tenant antigo, e (3) App Service sem acesso ao Azure Container Registry correto.

Parte 1 — Azure Portal
1

Configurar variáveis de ambiente no App Service

Sem isso o app não inicializa — banco, JWT e CORS ficam quebrados.

CRÍTICO

No Azure Portal, acesse seu App Service → Settings → Environment variables → aba App settings e adicione todas as variáveis abaixo:

Variável Valor esperado Impacto se ausente
DATABASE_URL String de conexão PostgreSQL completa App crasha ao iniciar
JWT_SECRET Chave secreta longa e aleatória 401 em todas as rotas — ninguém loga
FRONTEND_URL URL completa do App Service, ex: https://ctg-engineering.azurewebsites.net CORS bloqueia todos os requests do browser
NODE_ENV production Modo dev ativo, HTTPS não forçado
ADMIN_EMAIL E-mail do admin inicial Admin não é criado — não consegue logar
ADMIN_PASS Senha do admin (mín. 8 chars, maiúscula, número) Admin não é criado — não consegue logar
ADMIN_NAME Nome do admin, ex: Administrador CTG Opcional — usa "Administrador" como padrão
BACKEND_URL Mesma URL do FRONTEND_URL (app unificado) Opcional — afeta CSP headers
WEBSITES_PORT 3001 Opcional — Azure precisa saber a porta do container
⚠️
Se você usava o mesmo banco de dados (Railway PostgreSQL) da conta anterior, use a mesma DATABASE_URL e a mesma JWT_SECRET para que os tokens de usuários existentes continuem válidos. Se usar uma JWT_SECRET diferente, todos os usuários precisarão logar novamente.
ℹ️
Após salvar as variáveis, o Azure reinicia o container automaticamente. Aguarde 1–2 minutos antes de testar.
2

Verificar acesso do App Service ao Container Registry (ACR)

O App Service precisa conseguir baixar a imagem Docker do ACR.

CRÍTICO

No Azure Portal, acesse seu App Service → Settings → Deployment Center e confirme:

  • Registry está apontando para o ACR correto da conta corporativa (não o ACR pessoal antigo)
  • Image and tag está configurado corretamente (ctg-engineering:latest)
  • O App Service tem permissão de pull no ACR — ative a opção "Admin credentials" ou configure uma Managed Identity

Se precisar criar um novo ACR na conta corporativa, use o Azure CLI:

# Login na conta corporativa az login --tenant <ID-DO-TENANT-CORPORATIVO> # Criar ACR (se não existir) az acr create \ --resource-group <SEU-RESOURCE-GROUP> \ --name ctgengineeringacr \ --sku Basic \ --admin-enabled true # Ver as credenciais do ACR az acr credential show --name ctgengineeringacr
ℹ️
Anote o Login Server (ex: ctgengineeringacr.azurecr.io), o Username e uma das Passwords — você vai precisar deles no Passo 4 (GitHub Secrets).
Parte 2 — GitHub Actions (CI/CD)
3

Criar Service Principal no tenant corporativo

O GitHub Actions precisa de um "usuário de serviço" para fazer deploy no Azure da empresa.

CRÍTICO
🔴
O AZURE_CREDENTIALS que você usava pertence ao tenant pessoal. Ele não tem acesso ao tenant corporativo. É necessário criar um novo.

Execute no Azure CLI (com usuário da conta corporativa logado):

# 1. Obter o ID da sua subscription corporativa az account show --query id -o tsv # Copie o resultado — é o <SUBSCRIPTION-ID> # 2. Obter o ID do Resource Group onde está o App Service az group list --query "[].name" -o tsv # 3. Criar o Service Principal com acesso ao Resource Group az ad sp create-for-rbac \ --name "ctg-forecast-github-actions" \ --role contributor \ --scopes /subscriptions/<SUBSCRIPTION-ID>/resourceGroups/<RESOURCE-GROUP> \ --sdk-auth

O comando retorna um JSON como este — copie ele inteiro:

{ "clientId": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx", "clientSecret": "sua-chave-secreta-aqui", "subscriptionId": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx", "tenantId": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx", "activeDirectoryEndpointUrl": "https://login.microsoftonline.com", "resourceManagerEndpointUrl": "https://management.azure.com/", ... }
⚠️
Se o comando az ad sp create-for-rbac --sdk-auth retornar erro de permissão, peça ao administrador do tenant corporativo para executá-lo. É necessário permissão de Application Administrator no Azure AD.
4

Atualizar os GitHub Secrets do repositório

Substituir todas as credenciais que ainda apontam para a conta pessoal.

CRÍTICO

Acesse: GitHub → repositório → Settings → Secrets and variables → Actions e atualize (ou crie) os seguintes secrets:

Secret Valor a colocar
AZURE_CREDENTIALS O JSON completo gerado no Passo 3 (cole o bloco inteiro)
ACR_LOGIN_SERVER Ex: ctgengineeringacr.azurecr.io (obtido no Passo 2)
ACR_USERNAME Username do ACR (obtido no Passo 2)
ACR_PASSWORD Password do ACR (obtido no Passo 2)
ℹ️
O nome do App Service (ctg-engineering) está hardcoded no arquivo .github/workflows/azure-deploy.yml linha 9. Se o App Service na conta corporativa tem um nome diferente, atualize esta linha também.
Parte 3 — Executar o Deploy
5

Disparar o pipeline de deploy

Fazer o GitHub Actions buildar e publicar a imagem no novo ACR.

IMPORTANTE

Com os secrets atualizados, dispare o deploy de uma das formas abaixo:

Opção A — Manual pelo GitHub (recomendado para testar):

  • Acesse o repositório → aba Actions → selecione o workflow "Deploy CTG Engineering → Azure"
  • Clique em Run workflow → Run workflow (branch: main)
  • Acompanhe os logs em tempo real para ver se algum passo falha

Opção B — Via git push:

# Qualquer commit na branch main dispara o deploy automaticamente git commit --allow-empty -m "chore: trigger deploy to corporate Azure" git push origin main
⚠️
O pipeline faz: build Docker → push ACR → deploy App Service. Cada etapa leva ~3–5 minutos. Monitore os logs no GitHub Actions para identificar erros.

Erros comuns nos logs e o que fazer:

Erro no logCausaSolução
Login failed for ACR Credenciais ACR erradas Revisar ACR_USERNAME e ACR_PASSWORD
AZURE_CREDENTIALS is invalid Service Principal errado Regerar o SP no Passo 3
App Service not found Nome do App Service diverge Corrigir AZURE_WEBAPP_NAME no workflow
No access to resource group SP sem permissão no RG Recriar SP com o --scopes correto
Parte 4 — Verificação e Primeiro Acesso
6

Verificar se a aplicação subiu corretamente

Testar o health check antes de tentar logar.

VERIFICAR

Abra o browser ou use o curl e acesse o endpoint de saúde:

curl https://<SEU-APP>.azurewebsites.net/api/health # Resposta esperada: { "status": "ok", "version": "2.1.1-azure-ready" }
Resposta recebidaSignificadoO que fazer
✅ {"status":"ok"} App rodando normalmente Ir para o Passo 7
❌ Application Error Container não subiu Ver logs: App Service → Log stream
❌ 503 Service Unavailable App Service ainda iniciando Aguardar 2 minutos e tentar novamente
❌ Página do Azure padrão Imagem Docker não foi carregada Verificar Deployment Center e ACR

Para ver os logs em tempo real no Azure Portal:

  • App Service → Monitoring → Log stream
  • Procure por linhas como ✅ Usuário admin pronto ou erros como ❌ DATABASE_URL não definida
7

Fazer o primeiro login e validar o sistema

Verificar se o usuário admin foi criado e o sistema está funcional.

VALIDAR

Acesse a URL da aplicação e faça login com as credenciais definidas nas variáveis de ambiente:

Email: valor de ADMIN_EMAIL Senha: valor de ADMIN_PASS
  • Login funciona e redireciona para o dashboard
  • Usuários existentes (se banco é o mesmo) conseguem logar normalmente
  • Projetos e dados aparecem corretamente
  • Operações de criação/edição funcionam (sem erros 401/403)
  • Trocar a senha do admin após o primeiro acesso
✅
Se o banco de dados é o mesmo Railway PostgreSQL da conta anterior, todos os dados, usuários e projetos já existentes estarão disponíveis. Nenhuma migração de dados é necessária.
⚠️
Se usou uma JWT_SECRET diferente da anterior, os usuários já logados terão seus tokens invalidados e precisarão logar novamente. Isso é comportamento normal e esperado.
Situação Especial
8

Se o administrador do tenant corporativo precisar aprovar o SP

Alguns tenants corporativos restringem a criação de Service Principals.

SE NECESSÁRIO
⚠️
Em ambientes corporativos com Azure AD restrito, um Global Administrator ou Application Administrator do tenant pode precisar executar o comando do Passo 3 ou conceder a permissão a você.

Informações para passar ao admin de TI:

Preciso criar um Service Principal com as seguintes características: Nome: ctg-forecast-github-actions Role: Contributor Scope: /subscriptions/<ID>/resourceGroups/<RG-DO-APP-SERVICE> Finalidade: Deploy automatizado via GitHub Actions (CI/CD) Formato saída: --sdk-auth (JSON com credenciais)

Alternativamente, o admin pode criar o SP pelo Azure Portal: Azure Active Directory → App Registrations → New Registration → depois criar um Client Secret e atribuir a role Contributor no Resource Group.

📋 Resumo das ações

Passo 1 — CRÍTICO Configurar 5+ variáveis de ambiente no App Service (DATABASE_URL, JWT_SECRET, FRONTEND_URL, NODE_ENV, ADMIN_*)
Passo 2 — CRÍTICO Verificar que o App Service consegue acessar e baixar a imagem do ACR corporativo
Passo 3 — CRÍTICO Criar novo Service Principal no tenant corporativo via az ad sp create-for-rbac
Passo 4 — CRÍTICO Atualizar 4 GitHub Secrets: AZURE_CREDENTIALS, ACR_LOGIN_SERVER, ACR_USERNAME, ACR_PASSWORD
Passo 5 — Deploy Disparar o workflow manualmente ou via git push para buildar e publicar a nova imagem
Passos 6–7 — Validação Checar /api/health, fazer primeiro login como admin, validar dados e funcionalidades
