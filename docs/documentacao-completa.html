CTG Engenharia — Forecast System

Documentação completa do sistema de gestão de forecast de engenharia. Guia de arquitetura, manutenção e operação da aplicação.

1. Visão Geral do Projeto

O CTG Engenharia — Forecast System é uma aplicação web full-stack desenvolvida para a gestão e acompanhamento de forecast (previsão) orçamentário de projetos de engenharia da CTG Brasil. O sistema permite que engenheiros, coordenadores, gestores e administradores acompanhem budgets, forecasts, valores realizados (actual), metas e pools de investimentos de forma centralizada.

Funcionalidades Principais

  • Dashboard gerencial com KPIs consolidados, gráficos de S-Curve e visão por polo/usina
  • Forecast Wizard — assistente passo a passo para preenchimento mensal de forecast com importação SAP
  • Gestão de projetos — CRUD completo com atribuição de engenheiros
  • Chat por projeto — sistema de mensagens com controle de leitura
  • Consolidação anual — fechamento de ano com consolidação automática
  • Exportação Excel — relatórios individuais e consolidados via ExcelJS
  • Relatório HTML — geração de relatório mensal interativo a partir de planilha Excel
  • Controle de documentos — atas, cartas e relatórios técnicos com revisões
  • Gestão de férias — timeline visual por área com detecção de conflitos
  • Delegação de acesso — usuários podem delegar acesso a seus projetos
  • Sistema de feedback — canal de comunicação usuário → desenvolvedor
  • PWA — instalável como app nativo, com service worker e cache
  • Alertas inteligentes — mensagens não lidas, forecast vazio/stale, actual pendente, férias sem ADP

2. Arquitetura do Sistema

A aplicação segue uma arquitetura monolítica modular, onde o backend Express serve tanto a API REST quanto os arquivos estáticos do frontend (SPA). Em produção, tudo roda em um único container Docker.

Fluxo de Requisição

  1. Cliente (browser/PWA) faz requisição para /api/* ou qualquer rota SPA
  2. Trust Proxy configura app.set('trust proxy', 1) para obter IP real em ambientes com proxy (Azure/Railway)
  3. CORS valida origem contra FRONTEND_URL (em produção) ou permite tudo (dev)
  4. Security Headers (Helmet com CSP customizado) são aplicados
  5. Rate Limiter verifica limite por IP (apiLimiter: 150 req/min)
  6. Auth Middleware (nas rotas protegidas) valida JWT, revalida role/active no DB, aplica elevação por delegação
  7. Route Handler processa a lógica de negócio
  8. Response retorna JSON com status adequado

5. Backend — Entry Point (index.js)

O arquivo backend/src/index.js é o ponto de entrada da aplicação. Ele configura o servidor Express em uma cadeia ordenada de middlewares e rotas.

Ordem de Execução dos Middlewares

A ordem é criticamente importante — cada middleware depende do anterior:

  1. app.set('trust proxy', 1) — Configura o Express para confiar no primeiro proxy. Obrigatório em Azure/Railway para obter o IP real do cliente via x-forwarded-for.
  2. CORS — Primeiro middleware real. Configura origens permitidas dinamicamente via FRONTEND_URL. Em produção, valida origem; em dev, permite tudo. credentials: true para cookies.
  3. requireHTTPS — Redireciona HTTP → HTTPS em produção. Exceção para /api/health.
  4. securityHeaders — Helmet com Content Security Policy customizado (permite blob:, fonts.googleapis.com, cdnjs.cloudflare.com).
  5. cookieParser() — Parsing de cookies (necessário para JWT cookie ctg_token).
  6. compression() — Gzip compression de responses (melhora performance em ~60-80%).
  7. express.json({ limit: '10mb' }) — Parsing de JSON body com limite de 10MB (necessário para bulk upserts).

Health Check

A rota GET /api/health é registrada antes do rate limiter para garantir que monitors de saúde (Azure, uptime checks) nunca sejam bloqueados.

app.get('/api/health', (_, res) => {
  res.json({ status: 'ok', version: '2.1.1-azure-ready' });
});

Rate Limiter

Aplicado em /api — limita a 150 requisições por minuto por IP. Rotas específicas podem ter limites adicionais:

  • /api/export/* → heavyOpLimiter (10 req / 5 min)
  • /api/monthly-report/* → heavyOpLimiter (10 req / 5 min)
  • /api/auth/login → loginLimiter (10 req / 15 min)

Serving do SPA

Após as rotas de API, o Express serve arquivos estáticos de public/ (build do frontend). O catch-all app.get('*') retorna index.html para qualquer rota que não comece com /api, permitindo que o React Router gerencie as rotas client-side.

Inicialização

A função start() executa em sequência:

  1. initDB() — Inicializa o pool PostgreSQL e executa migrations (CREATE TABLE IF NOT EXISTS)
  2. seedAdmin() — Garante que o usuário admin exista (seed idempotente)
  3. app.listen(PORT) — Inicia o servidor (3001 em dev, 8080 em prod via Docker)

6. Backend — Banco de Dados

O banco de dados é PostgreSQL, acessado via pool nativo do pacote pg. O pool é criado uma única vez e exportado para todos os módulos.

Configuração do Pool

import pkg from 'pg';
const { Pool } = pkg;

export const pool = new Pool({
  connectionString: process.env.DATABASE_URL,
  ssl: { rejectUnauthorized: false }
});

O ssl: { rejectUnauthorized: false } é necessário para conexões com Railway/Azure onde o certificado pode não ser validado pelo CA bundle padrão.

Migrations

A função initDB() executa CREATE TABLE IF NOT EXISTS para cada tabela. Isso significa que:

  • A aplicação cria as tabelas automaticamente na primeira execução
  • Novas colunas/tabelas são adicionadas ao iniciar o app
  • Atenção: alterações em colunas existentes exigem migrações manuais (DROP + CREATE ou ALTER TABLE)
⚠️ Importante: Não há sistema de migrations versionadas (como Knex ou Prisma Migrate). Alterações no schema devem ser feitas manualmente com cuidado para não perder dados.

7. Backend — Autenticação & Autorização

JWT (JSON Web Token)

O sistema usa JWT assinada com jsonwebtoken. O token é armazenado em um cookie ctg_token com as flags httpOnly, secure (em produção) e sameSite: 'lax'.

// Assinatura do token
export function signToken(payload) {
  return jwt.sign(payload, process.env.JWT_SECRET, { expiresIn: '7d' });
}

// Payload típico
{
  userId: 1,
  email: 'usuario@ctgbrasil.com',
  role: 'engenheiro',
  name: 'João Silva'
}

Middleware requireAuth

Executa em 4 etapas:

  1. Extração do token — Tenta cookie (ctg_token), depois header Authorization: Bearer <token>
  2. Verificação — jwt.verify() valida assinatura e expiração
  3. Revalidação no DB — Busca usuário no banco para confirmar que ainda está ativo e com mesmo role
  4. Elevação por delegação — Se há delegação ativa recebida, eleva o role efetivo para o nível do delegador

Middleware requireRole(...roles)

Middleware de autorização que verifica se o req.user.role está na lista de roles permitidos:

// Exemplo de uso
router.post('/projects', requireRole('admin', 'gestor', 'planejador'), handler);

Middleware denyGerente

Bloqueia explicitamente usuários com role gerente de operações de escrita. Gerentes têm acesso somente leitura aos dados.

Middleware requireProjectAccess

Controle de acesso granular por projeto baseado no role:

RoleAcesso a Projetos
admin, gestor, planejador, gerenteTodos os projetos
coordenadorProjetos com engenheiros da sua área
engenheiroProjetos designados + delegações ativas

8. Backend — Segurança

Helmet com CSP Customizado

Helmet configura headers de segurança. A Content Security Policy (CSP) é customizada para permitir:

  • blob: — Necessário para URLs de objetos blob no frontend
  • fonts.googleapis.com — Google Fonts (DM Serif Display, Plus Jakarta Sans)
  • cdnjs.cloudflare.com — CDN para bibliotecas externas

Validação de Senha

Política mínima: 8 caracteres, 1 maiúscula, 1 minúscula, 1 número. Validada por validatePassword() no middleware validation.js.

A função sanitizeString() escapa HTML entities (< → &lt;, > → &gt;) em inputs de usuário para prevenir injeção de scripts.

API Client (utils/api.js)

import axios from 'axios';

const api = axios.create({
  baseURL: '/api',
  withCredentials: true  // Envia cookies automaticamente
});

// Interceptor: injeta Bearer token do localStorage
api.interceptors.request.use(config => {
  const token = localStorage.getItem('ctg_token');
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

// Interceptor: redirect para /login em 401
api.interceptors.response.use(
  res => res,
  err => {
    if (err.response?.status === 401) window.location.href = '/login';
    return Promise.reject(err);
  }
);
ℹ️ Nota: O withCredentials: true permite que cookies sejam enviados automaticamente. O interceptor de Bearer token serve como fallback para casos onde o cookie não está disponível (ex: desenvolvimento com proxy).

12. Frontend — Contextos

AuthContext (context/AuthContext.jsx)

Prove o estado global de autenticação para toda a aplicação:

  • user — Objeto do usuário logado (null se deslogado)
  • loading — Estado de carregamento durante verificação de auth
  • login(email, password) — POST /api/auth/login, salva token no localStorage
  • logout() — POST /api/auth/logout, limpa token e redireciona
  • fetchMe() — GET /api/auth/me, revalida role a cada 5 min (para delegações)

Hook useRole()

Retorna helpers booleanos baseados no role do usuário:

const { isAdmin, isGestor, isCoordenador, isEngenheiro, isPlanejador, isGerente, canEdit, canManage, canViewAll, userArea, hasAreaRestriction } = useRole();

SettingsContext (context/SettingsContext.jsx)

Prove configurações do sistema carregadas via GET /api/settings:

  • alert_stale_days — Dias para considerar forecast como "stale"

19. Docker

Dockerfile (Multi-Stage Build)

# Stage 1: Build do Frontend
FROM node:22-alpine AS frontend-build
WORKDIR /frontend
COPY frontend/package*.json ./
RUN npm ci
COPY frontend/ .
RUN npm run build

# Stage 2: Runtime
FROM node:22-alpine AS runtime
WORKDIR /app
COPY backend/package*.json ./
RUN npm ci --omit=dev
COPY backend/ ./
RUN mkdir -p public
COPY --from=frontend-build /frontend/dist ./public
ENV NODE_ENV=production
ENV PORT=8080
EXPOSE 8080
CMD ["node", "src/index.js"]

Explicação das Etapas

  1. Stage 1 (frontend-build) — Instala dependências do frontend e executa npm run build, gerando os arquivos estáticos em frontend/dist/
  2. Stage 2 (runtime) — Instala apenas dependências de produção do backend (--omit=dev), copia o código backend, e copia o build do frontend para public/

Vantagens do Multi-Stage Build

  • Imagem final menor — Dependências de dev do frontend não são incluídas
  • Segurança — Apenas pacotes de produção na imagem final
  • Self-contained — Backend + frontend em um único container

Execução Local com Docker

# Build da imagem
docker build -f backend/Dockerfile -t ctg-engineering .

# Execução (precisa de DATABASE_URL)
docker run -p 8080:8080 \
  -e DATABASE_URL=postgresql://user:pass@host:5432/db \
  -e JWT_SECRET=meu-segredo \
  ctg-engineering

Elevação por Delegação

Quando um usuário de nível superior delega acesso a outro, o role efetivo do delegado é elevado para o nível do delegador durante a delegação ativa. Isso permite que um engenheiro acesse projetos como se fosse um coordenador, por exemplo.

Regras de Auto-Registro

  • Roles permitidas no auto-registro: coordenador, engenheiro, gerente
  • Engenheiros e coordenadores exigem area (elétrica, mecânica, confiabilidade, coordenacao, modernizacao)
  • Novos registros ficam com status pending até aprovação pelo admin

24. Progressive Web App (PWA)

O frontend é configurado como PWA através do plugin vite-plugin-pwa.

Funcionalidades PWA

  • Service Worker — Gerado automaticamente pelo Workbox. Estrategia StaleWhileRevalidate para assets e NetworkFirst para API
  • Manifest — public/manifest.json com nome, ícones (192x192, 512x512), cores e modo de exibição (standalone)
  • Auto-update — Verifica nova versão a cada 60s. Exibe prompt com countdown de 30s para atualização
  • App Badge — navigator.setAppBadge(count) para mostrar contagem de alertas no ícone do app (mobile)
  • Instalação — O browser oferece instalar como app nativo (Android, iOS, Desktop)

Configuração do PWA (vite.config.js)

VitePWA({
  registerType: 'autoUpdate',
  manifest: { /* nome, ícones, cores */ },
  workbox: {
    globPatterns: ['**/*.{js,css,html,ico,png,svg,woff2}'],
    runtimeCaching: [
      { urlPattern: /^https:\/\/api\./i, handler: 'NetworkFirst' },
      { urlPattern: /^https:\/\/fonts\./i, handler: 'StaleWhileRevalidate' }
    ]
  }
})

25. Guia de Manutenção

Adicionando uma Nova Rota na API

  1. Crie o arquivo em backend/src/routes/nova-rota.js
  2. Use o padrão Express Router: 
    import express from 'express';
import { requireAuth } from '../middleware/auth.js';

const router = express.Router();

router.get('/', requireAuth, async (req, res) => {
  try {
    const { rows } = await pool.query('SELECT ...');
    res.json({ success: true, data: rows });
  } catch (err) {
    res.status(500).json({ success: false, error: err.message });
  }
});

export default router;
  3. Registre em backend/src/index.js: 
    import novaRotaRouter from './routes/nova-rota.js';
app.use('/api/nova-rota', requireAuth, novaRotaRouter);

Adicionando uma Nova Tabela ao Banco

  1. Adicione o CREATE TABLE IF NOT EXISTS em backend/src/db/schema.js dentro da função initDB()
  2. Execute a aplicação localmente para criar a tabela
  3. Em produção, a tabela será criada automaticamente no próximo deploy/restart
⚠️ Cuidado: Se precisar alterar uma coluna existente (ex: renomear, mudar tipo), o CREATE TABLE IF NOT EXISTS não fará nada. Será necessário: (1) Backup do banco, (2) DROP + CREATE da tabela, ou (3) ALTER TABLE manual.

Adicionando uma Nova Página no Frontend

  1. Crie o arquivo em frontend/src/pages/NovaPage.jsx
  2. Adicione a rota em frontend/src/App.jsx: 
    import NovaPage from './pages/NovaPage';
// Dentro do <Routes>:
<Route path="/nova" element={
  <RequireAuth>
    <NovaPage />
  </RequireAuth>
} />
  3. Adicione o link na Sidebar.jsx

Adicionando uma Nova Role/Permissão

  1. Adicione o novo role ao enum no schema.js (tabela users)
  2. Atualize auth.js — adicione o nível hierárquico
  3. Atualize AuthContext.jsx — adicione hook helper (isNovaRole)
  4. Atualize as validações nas rotas afetadas
